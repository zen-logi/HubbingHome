# Home Assistant / BroadLink 運用メモ

## 前提

Home Assistant は状態管理と自動化の中心、BroadLink は赤外線・RF 送信などの物理操作を担う。HubbingHome からは、可能な限り Home Assistant 経由で状態確認・操作を行い、BroadLink への直接操作は必要な範囲に限定する。

## 推奨構成

- Home Assistant を信頼できる宅内ネットワークまたは VPN 内に置く。
- HubbingHome から Home Assistant へは HTTPS または信頼できる内部経路で接続する。
- BroadLink デバイスは固定 IP または DHCP 予約を使用する。
- デバイス名、部屋名、エンティティ ID は設定で管理し、コードに直書きしない。

## Home Assistant 連携

- API トークンは長期アクセストークンを使用し、最小権限の専用ユーザーを用意する。
- REST API や WebSocket API の利用時はタイムアウト、リトライ上限、キャンセルを設定する。
- `entity_id` はサンプルでは `light.example_room` のようなダミー値にする。
- 状態取得とサービス呼び出しは別のクライアントメソッドとして設計する。
- サービス呼び出しは対象ドメイン、サービス名、対象エンティティをログに残す。ただし実住所や個人名を含む名称はマスク対象とする。

## BroadLink 連携

- BroadLink の IP アドレス、MAC アドレス、認証情報、学習済み IR/RF コードは秘密情報として扱う。
- IR/RF 送信は実機操作であり、テストやリトライで連続送信されないよう制限する。
- 学習済みコードはアプリケーション設定や DB に保存する場合も暗号化またはアクセス制限を検討する。
- 送信前に対象デバイス、コマンド種別、許可状態を検証する。
- ブロードキャスト探索は開発・初期設定用途に限定し、通常運用では固定設定を優先する。

## 操作安全性

- 電源、空調、ロック、加熱機器など生活安全に関わる操作は追加確認や許可リストを設ける。
- 同一コマンドの連続実行にはクールダウンを設ける。
- 自動化から呼ばれる操作は、手動操作よりも厳しい制限をかける。
- 失敗時は「再送するべきか」「状態確認だけに留めるべきか」を操作種別ごとに決める。

## 手動テスト手順の記録項目

- 実施日時
- 対象環境
- 対象 Home Assistant エンティティまたは BroadLink デバイス
- 実行した操作
- 期待結果
- 実結果
- 復旧手順
- 実機への副作用

## トラブルシュート観点

- Home Assistant の URL、トークン、証明書、到達性を確認する。
- エンティティ ID やサービス名が Home Assistant 側で変更されていないか確認する。
- BroadLink デバイスの IP アドレス、MAC アドレス、認証状態を確認する。
- 宅内ネットワークの VLAN、ファイアウォール、mDNS、ブロードキャスト制限を確認する。
- 同一操作が重複実行されていないか、相関 ID と操作ログで追跡する。
