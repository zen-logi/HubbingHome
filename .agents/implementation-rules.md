# 実装規約

## 目的

HubbingHome は Home Assistant や BroadLink などの宅内機器を扱うため、通常の Web/API 開発よりも誤操作・秘密情報漏えい・ローカル環境依存の影響が大きい。実装時は「安全に失敗する」「設定を外に出す」「実機操作は明示的に制限する」を基本方針とする。

## 作業範囲と衝突回避

- API/App 配下の実装担当がいる場合、共通規約・設定・文書の変更は `.agents` とルート設定を中心に行う。
- 既存変更は戻さない。差分があるファイルを触る前に内容を確認し、無関係な変更は維持する。
- 一時ファイルは可能な限り `D:\tmp` に置く。リポジトリ内に必要な場合は `.tmp` を使用し、成果物として残すべきもの以外はコミット対象にしない。
- 指示がない限り commit / PR は作成しない。

## コーディング規約

- .NET を使用する場合は .NET 10 以降を使用する。
- Microsoft および .NET の推奨する設計・実装を基本とする。
- API は MVC + Service 設計を基本とし、Controller から外部連携や状態更新を直接実装しない。
- Service は必ず Interface を継承し、DI で注入する。
- DI 登録とミドルウェア構成は `Startup.cs` に集約し、`Program.cs` は最小限にする。
- プライマリコンストラクタを適宜使用し、プライマリコンストラクタのパラメータを直接使用する。
- `.editorconfig` を参照し、C#、Razor、CSS、JSON の整形方針に合わせる。
- NuGet やフレームワークは必要に応じて導入し、それぞれの公式ベストプラクティスに従う。
- 設定値、トークン、ホスト名、MAC アドレス、デバイス ID、部屋名などの実環境情報はコードに直書きしない。
- 外部 API、Home Assistant、BroadLink、MQTT などの接続処理は、タイムアウト・リトライ上限・キャンセルを必ず持たせる。
- 実機を操作する処理は、読み取り系と書き込み系を明確に分ける。
- 書き込み系の操作には、ログ・監査に残せる操作名、対象、実行者または呼び出し元、結果を含める。
- デフォルト設定は実機に影響しない安全な値にする。サンプル設定も本物の値に見える内容は避ける。
- 例外は握りつぶさない。ユーザー向けレスポンスと内部ログで粒度を分ける。

## コメント規約

- C# の `summary` は必ず日本語で記述する。
- `summary` は「ですます」を使わず、末尾に句点を付けない。
- Interface を実装する Service の public method は `inheritdoc` を使用する。
- private method は日本語の `summary` を記述する。
- コメントおよび XAML/Razor 上の補足は日本語で記述する。ただしコンソール出力はこの限りではない。

## UI 規約

- クライアントは MAUI Blazor Hybrid を基本とし、UI は Razor コンポーネントで構成する。
- デザインは Tailwind CSS を使用する。
- shadcn/ui は React 前提のため直接採用しないが、低彩度の境界線、控えめな角丸、明確な押下状態、余白の一貫性を参考にする。
- iPad のタッチ操作を優先し、主要操作ボタンは十分な高さと押下領域を持たせる。

## 設定管理

- `appsettings.json` は非秘密の既定値のみを置く。
- 個人・家庭・実機固有の値は `appsettings.Development.json`、ユーザーシークレット、環境変数、または安全なシークレットストアで管理する。
- サンプル設定は `appsettings.Sample.json` のように、本物ではない値で構成する。
- 環境変数名は階層を `__` で表す形式を推奨する。

例:

```text
HubbingHome__HomeAssistant__BaseUrl
HubbingHome__HomeAssistant__AccessToken
HubbingHome__BroadLink__Devices__0__MacAddress
```

## テスト方針

- 自動テストは TUnit を使用する。
- 実機操作を伴うテストは通常の自動テストに含めない。
- Home Assistant / BroadLink 連携は、モックまたはスタブで契約テストを作る。
- 実機確認が必要なテストは手動テストとして分離し、対象デバイス、期待動作、復旧手順を記録する。
- セキュリティ関連の変更では、秘密情報がログ・例外・レスポンスに出ないことを確認する。

## ログ方針

- トークン、Cookie、Authorization ヘッダー、MAC アドレス、IP アドレス、位置情報、個人名、詳細な在宅状態はログにそのまま出さない。
- 操作ログは「何を試みたか」「成功/失敗」「相関 ID」を中心に残す。
- デバッグログに実環境情報が入りやすい箇所は、明示的にマスク処理を入れる。

## レビュー観点

- 実機へ意図せず命令が送られないか。
- 秘密情報や宅内情報がコード、設定、ログ、例外、テストデータに残っていないか。
- タイムアウトなしの外部通信がないか。
- 権限チェックなしで状態変更 API を呼べないか。
- サンプル設定がそのまま本番利用されても危険な操作につながらないか。
