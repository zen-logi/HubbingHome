# セキュリティ方針

## 基本方針

HubbingHome は宅内ネットワーク、スマートホーム機器、在宅状態に関わる情報を扱う。秘密情報だけでなく、機器名、部屋名、操作履歴、在宅・不在の推測につながる情報も保護対象として扱う。

## 保護対象

- Home Assistant の長期アクセストークン
- BroadLink デバイスの IP アドレス、MAC アドレス、認証情報、学習済み IR/RF コード
- MQTT、Webhook、外部連携の認証情報
- 住居、部屋、デバイス、住人、生活パターンを推測できる情報
- 操作履歴、センサー履歴、在宅・不在状態

## 禁止事項

- 秘密情報をリポジトリへコミットすること。
- 実機の IP アドレス、MAC アドレス、デバイス名をサンプル設定やテストデータへ入れること。
- Authorization ヘッダー、アクセストークン、Cookie をログ出力すること。
- 外部から到達可能なエンドポイントに認証なしの操作 API を置くこと。
- サンプル値として実在しそうなトークンや家庭内ネットワーク情報を使うこと。

## 認証・認可

- 状態参照と状態変更は権限を分ける。
- 実機操作、シーン実行、IR/RF 送信などの状態変更は認証必須にする。
- 管理者向け操作は通常ユーザー向け API と分離する。
- Webhook を受ける場合は署名検証、共有シークレット、送信元制限のいずれかを必須にする。

## ネットワーク

- Home Assistant や BroadLink への接続先は設定で指定する。
- `localhost` や宅内 IP 前提の処理でも、外部公開時に悪用されないよう認証・認可を実装する。
- SSRF を避けるため、任意 URL を受け取ってサーバー側からアクセスする設計は原則禁止する。
- 外部通信にはタイムアウトを設定する。

## 秘密情報の扱い

- 開発環境ではユーザーシークレットまたは環境変数を使用する。
- 本番環境では安全なシークレット管理機構を使用する。
- ログ、例外、画面表示、テレメトリには秘密情報を含めない。
- サンプル設定には `REPLACE_ME`、`example.local`、`00:00:00:00:00:00` などの明確なダミー値を使う。

## ログと監査

- 実機操作は操作種別、対象カテゴリ、結果、相関 ID を記録する。
- 個人や家庭を特定できる値はマスクする。
- 失敗時もトークンや詳細なヘッダーを出力しない。
- セキュリティイベントは通常ログと区別できるレベルまたはカテゴリで記録する。

## インシデント対応

秘密情報が漏えいした可能性がある場合は、次の順で対応する。

1. 該当トークンや認証情報を無効化・再発行する。
2. リポジトリ、ログ、成果物、CI 出力から露出範囲を確認する。
3. 影響を受ける Home Assistant / BroadLink / 外部連携の接続設定を更新する。
4. 再発防止として設定管理、ログマスク、レビュー観点を修正する。

## レビュー時の確認リスト

- 秘密情報が差分に含まれていない。
- サンプル値が明確なダミーである。
- 状態変更 API が認証・認可を通る。
- 実機操作にタイムアウトと失敗処理がある。
- ログに家庭内情報やトークンが出ない。
